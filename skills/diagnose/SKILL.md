通过解析症状 → 代码调研 → 主动复现 → 根因定位，自主生成结构化缺陷报告。

## 核心理念

用户报 bug 时说的往往是**表面症状**而非**根本原因**。诊断的目标是：

1. **解析**：从用户描述中提取症状、触发条件、期望行为
2. **调研**：沿代码路径追踪，缩小嫌疑范围
3. **复现**：如果是 UI 相关问题，用 Playwright 复现并截图取证
4. **定位**：从症状追溯到根因，找到具体代码位置
5. **报告**：输出标准化缺陷报告，含复现证据和修复方向

与 `requirement` 的区别：`requirement` 是需求链路的委托书（"要造什么"），`diagnose` 是缺陷链路的起诉书（"哪里坏了"）。

## Phase 0: 解析症状

### 目标
从用户输入中提取结构化的缺陷初判，不追问，直接推导。

### 操作

1. **通读用户输入**，识别：
   - 表面症状（用户能观察到什么）
   - 用户的猜测/归因（可能有误，但提供线索）
   - 已尝试的排查
   - 如果用户提供了截图，用 `Read` 查看

2. **自主推导**：
   - **复现路径**：从用户描述推断操作步骤
   - **触发条件**：每次都出现还是偶发？特定数据/配置才触发？
   - **期望 vs 实际**：正确行为是什么？实际看到什么？
   - **数据依赖**：复现需要特定的数据吗？

## Phase 1: 代码调研与根因定位

### 目标
沿代码路径追踪，缩小嫌疑范围，定位根因。

### 操作

1. **沿症状追踪代码路径**
   - 从用户描述的页面/功能出发，找到对应的代码入口
   - 使用 `Agent`（subagent_type=Explore）或 `Grep`/`Read` 调研
   - 沿调用链追踪：组件 → hooks → API → 数据层

2. **关注维度**
   - **数据流**：数据从哪来、经过哪些变换、到哪去
   - **状态管理**：涉及哪些状态？转换是否有遗漏？
   - **边界条件**：空值、并发、竞态、类型不匹配
   - **布局问题**：flex/grid 容器、overflow、高度约束
   - **近期改动**：`git log --oneline -20 -- <相关文件>`

3. **收敛根因**
   - 列出 2-3 个最可能的根因假设
   - 结合代码证据验证假设
   - 确定根因后定位到具体文件:行号

## Phase 2: 主动复现（可选）

如果是 UI/交互相关的 bug 且 Playwright 工具可用，主动复现取证。如果是纯逻辑/后端问题，可跳过此 Phase。

### 操作

1. **导航到起始页面**
   ```
   mcp__playwright__browser_navigate → 目标 URL
   ```

2. **按复现步骤逐步操作**
   - 每一步先 `browser_snapshot` 获取页面结构
   - 用 `browser_click` / `browser_type` / `browser_fill_form` 执行操作
   - 关键步骤前后用 `browser_take_screenshot` 截图
   - 等待页面响应：`browser_wait_for`

3. **Bug 触发后取证**
   - 截图记录 bug 现象：`browser_take_screenshot`
   - 如果是控制台报错：`browser_console_messages`
   - 如果是网络请求问题：`browser_network_requests`
   - 如果需要检查元素状态：`browser_evaluate` 执行 JS 检查 DOM

### 截图管理
截图统一放在报告同目录的 `DEFECT.assets/` 子目录下：
```
{报告目录}/
├── DEFECT.md
└── DEFECT.assets/
    ├── diagnose-{简述}-step{N}.png
    └── diagnose-{简述}-bug.png
```
报告中用相对路径引用：`![desc](DEFECT.assets/xxx.png)`

## Phase 3: 输出缺陷报告

### 输出位置

如果项目中有约定的文档目录（如 `.task/`、`docs/`、`bugs/`），使用该目录。否则写入项目根目录 `DEFECT.md`。

### 四不可约元素

报告围绕四个不可约元素组织——缺少任何一个都无法完成修复：

```
           缺陷报告
              │
    ┌─────────┼─────────┐
    │         │         │
  Delta    Path     Location
  (what)   (how)    (where)
    │         │         │
    └─────────┼─────────┘
              │
           Impact
           (so what)
```

1. **Delta（偏差）**——"错在哪"：期望行为 + 实际行为 + 二者之差
2. **Path（复现路径）**——"怎么触发"：环境/上下文 + 操作步骤 + 输入数据
3. **Location（定位）**——"在哪里"：模块/功能/页面 + 具体代码坐标 + 根因分析
4. **Impact（影响）**——"多严重"：严重度 + 影响范围

### 报告自检清单

- [ ] **Delta**：期望和实际都写了吗？偏差描述清晰吗？
- [ ] **Path**：任何人按步骤都能复现吗？
- [ ] **Location**：定位到具体文件:行号了吗？根因分析了吗？
- [ ] **Impact**：严重度和影响范围都评估了吗？

### 流程

1. 自主完成 Phase 0-2
2. 生成报告，用 `Write` 写入约定路径
3. 向用户展示报告摘要，告知文件位置

## 执行规则

1. **不追问，自主推导**：从用户输入 + 代码调研中提取全部信息，直接输出报告
2. **代码先行**：先读代码定位，再决定是否需要 Playwright 复现
3. **截图取证**：UI 相关 bug 必须截图（如 Playwright 可用）
4. **收敛而非发散**：每一步都在缩小嫌疑范围
5. **根因优先**：不要停在症状描述，必须追到代码级根因
6. **过程备注**：执行过程中遇到重试、惊讶、绕路等偏差信号时，记录到报告的「过程备注」节。格式：`[重试/惊讶/绕路/确认/环境] 简述`
