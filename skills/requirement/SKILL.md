通过代码调研 → 方案收敛，自主生成结构化需求报告。

## 核心理念

需求报告是**委托书**——指着一片空白说"这里要造一个东西"。

用户说出来的往往是**解决方案**而非**问题本身**。你的任务是穿透表面诉求，从 Why 出发，结合代码调研自主重建完整的需求画面。

- 缺陷报告描述**已存在的偏差**，需求报告描述**尚不存在的能力**
- 缺陷报告的难点是**精确描述**，需求报告的难点是**认知对齐**
- 缺陷报告有隐式锚点（修复前的正确行为），需求报告没有锚点——"做到了"的定义必须显式声明

## 链路定位

```
需求报告 ──→ 实现 ──→ 验收
 委托书     施工方案    竣工验收

 Who    ──→  User Flow   ──→  场景走查
 Why    ──→  Design Why  ──→  动机满足度
 What   ──→  Change Set  ──→  能力核对
 Accept ──→  Test Cover  ──→  清单逐项 ✓/✗
 Constr ──→  Compliance  ──→  约束逐项 ✓/✗
```

## Phase 0: 解析用户意图

### 目标
从用户输入中提取完整的需求画面，不追问，直接推导。

### 操作

1. **通读用户输入**，识别：
   - 表面诉求（用户直接说要做什么）
   - 隐含动机（为什么要做）
   - 用户的方案设想（如果有）
   - 约束线索（时间、兼容性、不可改变的部分）

2. **自主推导五元素**：
   - **Who**：从用户描述和项目上下文推断使用者角色和场景
   - **Why**：从表面诉求倒推动机——用户为什么要这个？当前痛点是什么？
   - **What**：将用户的方案设想提升到能力声明层——不锁定实现，只定义能力边界

## Phase 1: 代码调研

### 目标
理解项目现状，为定义 What、Acceptance、Constraint 提供基础。

### 操作

1. **调研现有实现**
   - 使用 `Agent`（subagent_type=Explore）或 `Grep`/`Read` 调研相关功能区域
   - 了解：现有数据模型、相似功能的交互模式、当前 UX/DX
   - 识别：可复用的模式、潜在冲突、会受影响的现有行为

2. **关注维度**
   - **UX 现状**：当前用户操作路径、界面布局
   - **DX 现状**：对外接口、组件 API、配置项
   - **数据模型**：相关的 DB schema、数据流
   - **影响面**：如果改了这块，哪些地方会受影响？

3. **用调研修正推导**
   - 代码现状可能推翻或修正 Phase 0 的初步推导
   - 发现用户未提及的约束、依赖、兼容性问题

## Phase 2: 方案收敛与验收定义

### 目标
确定 What 的精确边界，定义 Acceptance 清单和 Constraint 红线。

### 操作

1. **收敛 What**
   - 基于调研确定最合理的方案路径
   - 粒度把控：能力声明层，不锁定 UI/实现细节

2. **定义 Acceptance（验收标准）**
   - 从 What 推导可判定的命题集
   - 每条 Acceptance 必须是 ✓/✗ 可判定的——不用"应该好用"这种模糊描述

3. **明确 Constraint（约束）**
   - **业务约束**：从用户描述中提取不可违反的规则
   - **技术约束**：从代码调研中提取——架构限制、依赖关系、影响面
   - **兼容约束**：现有行为哪些必须保持不变？

## Phase 3: 输出需求报告

### 输出位置

如果项目中有约定的文档目录（如 `docs/`、`.task/`、`requirements/`），使用该目录。否则写入项目根目录 `REQ.md`。

### 五不可约元素

报告围绕五个不可约元素组织——缺少任何一个都无法闭合链路：

```
           需求报告
              │
      ┌───────┼───────┐
      │       │       │
    Who     Why     What
   (谁)    (为何)   (做什么)
      │       │       │
      └───────┼───────┘
              │
      ┌───────┴───────┐
      │               │
 Acceptance       Constraint
 (怎么算成)       (什么不行)
```

1. **Who（主体 + 场景）**——"谁在什么处境下"
2. **Why（动机）**——"为什么需要这个能力"（根节点，不可从其他元素推导）
3. **What（能力声明）**——"系统要获得什么新能力"（能力边界层，不锁定实现）
4. **Acceptance（验收标准）**——"怎么判定做到了"（可判定命题集，只有 true/false）
5. **Constraint（约束）**——"什么边界不可越过"（事前红线，不是事后影响描述）

### 报告模板

```markdown
# 需求报告：{一句话标题}

> 创建时间：{YYYY-MM-DD HH:mm}

## 1. Who（主体 + 场景）

### 使用者
{谁会用这个功能——角色/身份}

### 使用场景
{什么处境下需要——频率、触发时机、当前解法}

## 2. Why（动机）

### 痛点
{当前遇到什么问题 / 缺失什么能力}

### 做了的价值
{解决什么问题、带来什么好处}

### 不做的代价
{维持现状会怎样——能忍受的不便 / 持续恶化的问题 / 阻断性影响}

## 3. What（能力声明）

### 核心能力
{系统要获得什么新能力——能力边界层描述，不锁定 UI/实现细节}

- {能力 1}
- {能力 2}
- ...

### 不做（Out of Scope）
- {明确排除的内容}

## 4. Acceptance（验收标准）

{可判定的命题清单——每条只有 ✓/✗，无模糊地带}

- [ ] {Given 前置条件, When 操作, Then 预期结果}
- [ ] ...

## 5. Constraint（约束）

### 业务约束
- {不可违反的业务规则}

### 技术约束
- {架构限制、性能预算、兼容性要求——来自代码调研}

### 不可打破的现有行为
- {必须保持不变的现有功能/接口}

## 参考
- {相关代码路径、可复用模块、规范文档——从代码调研中提取}

## 过程备注

{执行过程中捕获的学习信号。无则留空}
```

### 报告自检清单

生成报告后，对照五元素逐项检查：

- [ ] **Who**：使用者和场景都明确了吗？
- [ ] **Why**：痛点清晰吗？"做了/不做"的价值和代价都写了吗？
- [ ] **What**：能力声明在正确的粒度吗？（不是模糊轮廓，也不是锁定实现）
- [ ] **Acceptance**：每条验收标准都是 ✓/✗ 可判定的吗？覆盖了 What 声明的每项能力吗？
- [ ] **Constraint**：技术约束有从代码调研中提取吗？

### 流程

1. 自主完成 Phase 0-2
2. 生成报告，用 `Write` 写入约定路径
3. 向用户展示报告摘要，告知文件位置

## 执行规则

1. **不追问，自主推导**：从用户输入 + 代码调研中提取全部信息，直接输出报告
2. **Why 是北极星**：所有推导最终要收敛到 Why——它决定了其他四个元素的形态
3. **代码先行**：能通过读代码推断的就不猜测
4. **粒度校准**：What 保持在能力声明层——不是模糊轮廓，也不锁定实现
5. **Acceptance 必须可判定**：模糊的验收标准等于没有标准
6. **过程备注**：执行过程中遇到重试、惊讶、绕路等偏差信号时，记录到报告的「过程备注」节。格式：`[重试/惊讶/绕路/确认/环境] 简述`
